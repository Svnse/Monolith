# Turn Trace Spec v1 — Monolith front-half causal visibility

**Status:** spec, no code. Implementation gated on user approval of design + open questions resolved.

## 0. Frame

The advanced Monolith's `TurnContextTrace` (`core/turn_context_trace.py`, 218 lines) is the model — but it ports poorly because it assumes a frame compiler with item-level ranking that Restore doesn't have. This spec scales the same discipline to Restore's substrate and ships **producer + consumer in the same change** so we don't reproduce the bifurcated-orphan pattern.

**What this is (the front half):**
- **Layer A — Stage Trace**: persisted record of what each pipeline stage *did* on a turn (which interceptors ran, what they injected, what they dropped, why).
- **Layer B — Frame Trace**: persisted record of what the LLM *saw* on a turn (the final assembled message list, sizes, backend dispatched).
- **Layer C — Inspector Query**: a joined query API that returns (A + B) for a given turn, plus a `/trace` slash command and an `inspect_trace` tool the model can call to self-inspect.

**What this isn't (the back half — explicitly out of scope):**
- Post-turn outcome trace (did the user accept / correct / redo?)
- Verifier trace (did the response pass deterministic verification? — Restore has no verifier yet anyway)
- Memory-write trace (did this turn write to ACU / continuity / archive?)
- Replay primitive (re-run the assembler with tweaked inputs against a stored trace)

**Why now:** the substrate is now complex enough that "why did the model see X" takes more than 2 minutes to reason about. Pipeline is `continuity → effort → context_refresh → adaptive_budget` with conditional fires per interceptor; each interceptor can inject ephemeral messages, mutate config, or skip. Without trace, debugging "the model didn't see continuity" requires re-running with print statements.

**Why this scope:** the user named this as "the missing front half" of a causal loop. Layers A + B + C are necessary and sufficient for **input-side causality** (what ran, what was seen). Outcome/verifier/memory linkages require subsystems that don't exist in Restore today — adding them now would be vertical without horizontal. Layer C's join key (`turn_id`) is preserved so the back half attaches without schema migration.

## 1. Architecture overview

```
USER MESSAGE
   │
   ▼ apply_interceptors(messages, config)
   │   ┌─ Layer A: each interceptor emits a StageTraceRecord ─┐
   │   │                                                       │
   │   ▼                                                       │
   │  continuity_interceptor    → record (kept/skipped, items_added, ...)
   │   │
   │   ▼
   │  effort_interceptor        → record
   │   │
   │   ▼
   │  context_refresh_interceptor → record
   │   │
   │   ▼
   │  adaptive_budget_interceptor → record (currently parked but recorded)
   │
   ▼ engine/llm.py:_compile_system_prompt
   │   ┌─ Layer A continued: each prompt stage emits a StageTraceRecord ─┐
   │   │                                                                  │
   │   ▼
   │  master_prompt        → record (chars added)
   │  skills_catalog       → record
   │  [ENVIRONMENT]        → record
   │  [IDENTITY]           → record
   │  [WORLD STATE]        → record (snapshot taken)
   │  [THINKING] suppress  → record (conditional)
   │
   ▼ engine/llm.py:generate (before backend dispatch)
   │   ┌─ Layer B: FrameTraceRecord emitted with final messages snapshot ─┐
   │
   ▼ backend (gguf / gguf_api / openai-compat)
   │
LLM RESPONSE

(post-turn / outcome / verifier / memory writes — back half — NOT IN SCOPE)
```

**Storage:** dedicated SQLite at `LOG_DIR / "turn_trace.sqlite3"`. Two tables — `stage_traces` and `frame_traces` — joined on `turn_id`. Separate from `EventLedger` (already classified bifurcated; not adding writes there).

**Join key:** `turn_id` = the `Task.id` UUID generated by `Task.new()` in `core/task.py:28`. Already exists per-turn; no new identity primitive required. Engine-side stages may emit before the task's outer turn is finalized, but they share the same `task_id` because the engine's `_current_task_id` field (`engine/llm.py:679`) propagates it.

## 2. Layer A — Stage Trace

### Schema

```python
@dataclass(frozen=True)
class StageTraceRecord:
    turn_id: str                    # Task.id UUID, join key
    seq: int                        # ordinal within turn (0,1,2,...)
    stage_name: str                 # "continuity_interceptor", "system_prompt:world_state", ...
    stage_kind: str                 # "interceptor" | "prompt_stage"
    entered_at: str                 # ISO-8601
    exited_at: str                  # ISO-8601
    outcome: str                    # "ran" | "skipped" | "errored"
    outcome_reason: str | None      # human-readable for skipped/errored, None for ran
    messages_in: int                # message count entering stage (interceptors)
    messages_out: int               # message count after stage
    items_added: tuple[StageItem, ...]      # what got injected
    items_dropped: tuple[StageItem, ...]    # what got removed (rare in Restore today)
    metadata: dict[str, Any]        # stage-specific (e.g. {"effort_tier": "high"})
```

```python
@dataclass(frozen=True)
class StageItem:
    kind: str                       # "ephemeral_user" | "system_prompt_chunk" | "config_mutation"
    content_hash: str               # sha256 first 12 chars of content
    content_preview: str            # first 280 chars, single-line
    source: str                     # "continuity" | "effort" | "world_state" | ...
    drop_reason: str | None         # set on items_dropped only — "duplicate"|"flag_off"|"empty"|"errored"
```

**Invariants** (enforced in `__post_init__`):
- `outcome ∈ {"ran","skipped","errored"}`.
- `outcome="skipped"` or `"errored"` requires `outcome_reason` non-empty.
- `outcome="ran"` requires `outcome_reason` is `None`.
- Items in `items_added` have `drop_reason=None`; items in `items_dropped` have `drop_reason` non-empty.
- `entered_at <= exited_at`.

### Producers

**Pre-LLM interceptors** (in `core/message_interceptors.py:apply_interceptors`):
- Wrap each interceptor call with a stage-record builder. The wrapper captures `(messages_in, messages_out, items_added)` by diff. Errors are caught, recorded as `outcome="errored"`, and re-raised (interceptor errors stay loud — record is best-effort observation).
- Producers per interceptor: `continuity_interceptor`, `effort_interceptor`, `context_refresh_interceptor`, `adaptive_budget_interceptor`.

**Prompt-assembly stages** (in `engine/llm.py:_compile_system_prompt`):
- Six stages — `master_prompt`, `skills_catalog`, `environment`, `identity`, `world_state`, `thinking_suppress`.
- Each emits a record with `stage_kind="prompt_stage"`, `messages_in=0/messages_out=0` (these stages mutate string, not message list), and `items_added` carrying the chunk that got appended.

### Storage

```sql
CREATE TABLE IF NOT EXISTS stage_traces (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    turn_id TEXT NOT NULL,
    seq INTEGER NOT NULL,
    stage_name TEXT NOT NULL,
    stage_kind TEXT NOT NULL,
    entered_at TEXT NOT NULL,
    exited_at TEXT NOT NULL,
    outcome TEXT NOT NULL,
    outcome_reason TEXT,
    messages_in INTEGER NOT NULL,
    messages_out INTEGER NOT NULL,
    items_added_json TEXT NOT NULL DEFAULT '[]',
    items_dropped_json TEXT NOT NULL DEFAULT '[]',
    metadata_json TEXT NOT NULL DEFAULT '{}'
);
CREATE INDEX IF NOT EXISTS idx_stage_turn ON stage_traces(turn_id, seq);
CREATE INDEX IF NOT EXISTS idx_stage_time ON stage_traces(entered_at);
```

### Per-spec template

| Question | Answer |
|---|---|
| **Closed-loop host** | Layer C inspector query — `/trace` slash command + `inspect_trace` tool. Writes ship in the same change as the consumer. |
| **Authority tier** | Pure diagnostic. Records are observations of behavior, never the source of behavior. Cannot influence any other subsystem. |
| **Evidence channel** | SQLite table, indexed on `turn_id` and `entered_at`. Append-only during a turn. |
| **Unlock criterion** | Always-on at production. Flag `MONOLITH_TURN_TRACE_V1` (default ON) for safe rollback. |
| **Retire/demote** | Retention policy — TTL of 30 days (configurable). `DELETE FROM stage_traces WHERE entered_at < ?` runs on a 24h scheduled daemon, same shape as world_state flush. |
| **Replayability** | Records are deterministic given the message diff. Trace + final messages snapshot = enough to re-run the assembler with the same inputs. (Replay primitive itself is out of scope; trace is the *substrate* for replay.) |

## 3. Layer B — Frame Trace

### Schema

```python
@dataclass(frozen=True)
class FrameTraceRecord:
    turn_id: str                    # Task.id UUID, join key
    captured_at: str                # ISO-8601
    backend: str                    # "gguf" | "gguf_api" | "openai" | "vllm" | "sglang"
    engine_key: str                 # "llm_<uuid>" — per-session engine
    gen_id: int                     # EngineBridge._gen_id at dispatch time
    final_messages: tuple[FrameMessage, ...]
    system_prompt_chars: int        # size of stage-0 message (or assembled system)
    user_prompt_chars: int          # size of latest non-ephemeral user message
    total_chars: int                # sum across messages
    config_snapshot: dict[str, Any] # {"max_tokens": ..., "temp": ..., "thinking": ..., "effort_tier": ...}
    metadata: dict[str, Any]        # backend-specific (model name, ctx_limit, ...)
```

```python
@dataclass(frozen=True)
class FrameMessage:
    role: str                       # "system" | "user" | "assistant" | "tool"
    content_hash: str               # sha256 first 12 chars
    content_preview: str            # first 280 chars
    content_chars: int              # full size
    ephemeral: bool                 # True for interceptor-injected
    source: str | None              # "continuity" | "effort" | "user" | None for assistant
```

**Note on content storage:** `content_preview` is bounded; `content_hash` lets a future replay primitive verify identity against an external archive. **Full content is NOT stored in Layer B by default** to bound storage. If the user wants full content, an opt-in flag `MONOLITH_TURN_TRACE_FULL_CONTENT` writes to a separate `frame_full_content` table — but this is a privacy / disk-space trade-off and starts off.

### Producer

**Single emit point**: `engine/llm.py:LLMEngine.generate`, immediately before backend dispatch (after `apply_interceptors`, after `_compile_system_prompt`). One record per generation. Tool followups (which are separate generations) get separate records — they share the outer-turn `task_id` if the followup carries it forward; otherwise each followup gets its own `turn_id`.

### Storage

```sql
CREATE TABLE IF NOT EXISTS frame_traces (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    turn_id TEXT NOT NULL,
    captured_at TEXT NOT NULL,
    backend TEXT NOT NULL,
    engine_key TEXT NOT NULL,
    gen_id INTEGER NOT NULL,
    system_prompt_chars INTEGER NOT NULL,
    user_prompt_chars INTEGER NOT NULL,
    total_chars INTEGER NOT NULL,
    final_messages_json TEXT NOT NULL DEFAULT '[]',
    config_snapshot_json TEXT NOT NULL DEFAULT '{}',
    metadata_json TEXT NOT NULL DEFAULT '{}'
);
CREATE INDEX IF NOT EXISTS idx_frame_turn ON frame_traces(turn_id);
CREATE INDEX IF NOT EXISTS idx_frame_time ON frame_traces(captured_at);
```

### Per-spec template

| Question | Answer |
|---|---|
| **Closed-loop host** | Layer C inspector query. |
| **Authority tier** | Pure diagnostic. Snapshot of what the LLM saw; never modifies. |
| **Evidence channel** | SQLite table, indexed on `turn_id` and `captured_at`. |
| **Unlock criterion** | Always-on at production; same `MONOLITH_TURN_TRACE_V1` flag as Layer A. |
| **Retire/demote** | Same 30-day TTL daemon as Layer A. |
| **Replayability** | Hash-keyed; full content opt-in via secondary table. The hash + Layer A's items_added let future replay reconstruct the assembled frame without storing full content by default. |

## 4. Layer C — Inspector Query

### Public API

```python
# core/turn_trace.py — query module

def get_turn_trace(turn_id: str) -> TurnTraceJoined | None:
    """Return joined Layer A + Layer B for a turn. None if turn not found."""

def list_recent_turns(limit: int = 20) -> list[TurnTraceSummary]:
    """Return summaries of recent turns, newest first."""

def search_turns(
    *, since: str | None = None,
    until: str | None = None,
    backend: str | None = None,
    has_errored_stage: bool | None = None,
) -> list[TurnTraceSummary]:
    """Filter recent turns. All filters AND-combined."""
```

```python
@dataclass(frozen=True)
class TurnTraceJoined:
    turn_id: str
    stages: tuple[StageTraceRecord, ...]   # Layer A, ordered by seq
    frame: FrameTraceRecord | None         # Layer B, None if generation didn't complete
    summary: dict[str, Any]                # {stage_count, errored_stages, total_chars, ...}

@dataclass(frozen=True)
class TurnTraceSummary:
    turn_id: str
    captured_at: str
    backend: str
    stage_count: int
    errored_stage_count: int
    total_chars: int
    frame_present: bool
```

### Slash command surface

- `/trace` — print summary of last 5 turns (turn_id, time, stage count, errored count, total chars).
- `/trace <turn_id>` — print joined record for that turn.
- `/trace last` — alias for the most recent turn.
- `/trace errors` — list recent turns with `errored_stage_count > 0`.

### Tool surface (model-callable)

```json
{"name":"inspect_trace","arguments":{"turn_id":"<uuid>"}}
{"name":"inspect_trace","arguments":{"verb":"recent","limit":5}}
{"name":"inspect_trace","arguments":{"verb":"errors"}}
```

The tool is read-only. Returns the joined record as JSON. Schema added to `BUILTIN_TOOL_SCHEMAS`. Lives as an executor in `core/skill_runtime.py` (builtin), not a dynamic skill — it's infrastructure.

### Per-spec template

| Question | Answer |
|---|---|
| **Closed-loop host** | The user (`/trace` command) and the model (`inspect_trace` tool). Both are real consumers; this is what closes the loop on Layers A + B. |
| **Authority tier** | Read-only diagnostic. Cannot mutate trace data; cannot trigger any other subsystem. |
| **Evidence channel** | Returns structured records to caller. Slash command renders to chat trace; tool returns to skill_runtime envelope. |
| **Unlock criterion** | Always-on. Same flag as Layers A/B — if writes are off, queries return empty results gracefully. |
| **Retire/demote** | Pure read; nothing to retire. |
| **Replayability** | Yes — the joined record is the input to a future replay primitive (`replay_turn(turn_id)` would be the natural extension). |

## 5. Back-half hooks (out of scope; preserved)

The user named the missing back half: outcome / verifier / memory-write linkage. This spec preserves the join key (`turn_id`) so the back half attaches without schema migration.

Future tables (NOT to be created in this spec):

```sql
-- (Future) Outcome trace: did the user accept/correct/redo?
CREATE TABLE outcome_traces (
    turn_id TEXT NOT NULL,
    user_response_kind TEXT,    -- "accepted" | "corrected" | "ignored" | "redone"
    user_response_at TEXT,
    correction_diff TEXT
);

-- (Future) Verifier trace: did the response pass deterministic checks?
CREATE TABLE verifier_traces (
    turn_id TEXT NOT NULL,
    verifier_name TEXT NOT NULL,
    passed INTEGER NOT NULL,
    findings_json TEXT
);

-- (Future) Memory-write trace: what did this turn write to ACU/continuity/archive?
CREATE TABLE memory_write_traces (
    turn_id TEXT NOT NULL,
    target_store TEXT NOT NULL,    -- "acu" | "continuity" | "chat_archive" | "scratchpad"
    op TEXT NOT NULL,              -- "insert" | "update" | "retire"
    target_id TEXT,
    written_at TEXT
);
```

When any of these land, `get_turn_trace(turn_id)` extends to return the additional joined data. Layer C's API contract changes by adding optional fields, not by replacing schema.

**Invariant to preserve through back-half work:** every back-half table joins on `turn_id` and only `turn_id`. Don't introduce a new join key. The join key is `Task.id` from the day Layer A ships.

## 6. Storage choice

**Picked:** dedicated SQLite at `LOG_DIR / "turn_trace.sqlite3"`.

**Why not EventLedger** (`core/event_ledger.py`): EventLedger is already classified bifurcated in the spirit of the synergy spec — writes happen, no production reader exists. Adding more writes there compounds the problem. Layer C is the consumer; trace data should live in a store the consumer queries.

**Why not world_state.json**: world_state is a single-document store; trace is multi-row temporal data. Wrong shape.

**Why not per-turn JSON files**: queryability matters for `search_turns` and `/trace errors`. SQLite indexes solve this; flat files don't.

**Schema migration discipline:**
- v1 ships with the two tables above.
- New columns added via `ALTER TABLE ... ADD COLUMN ... DEFAULT ...` (same pattern as `EventLedger._ensure_schema` at `core/event_ledger.py:39-62`).
- Back-half tables added by `CREATE TABLE IF NOT EXISTS` — no migration needed.
- A `schema_version` table tracks current version; producer code refuses to write if version mismatches.

**Retention:** 30-day TTL by default, configurable via `MONOLITH_TURN_TRACE_TTL_DAYS`. A scheduled daemon at 24h cadence runs `DELETE FROM <table> WHERE captured_at < ?`. Bootstrap registers the daemon if `MONOLITH_TURN_TRACE_V1` is on.

## 7. Open questions (resolve before implementation)

1. **Turn ID propagation for prompt-assembly stages.** Interceptors receive `(messages, config)`. The `task_id` exists in the engine but isn't currently in `config`. Two options: (a) bootstrap injects `_turn_id` into config before `apply_interceptors`; (b) `apply_interceptors` signature changes to `(messages, config, turn_id)`. (a) is less invasive; (b) is more explicit. **Default: (a).**

2. **Tool followup turns.** Each tool followup is a separate `generate` task with a new `task_id`. Do followups share the *outer* turn_id (better for the inspector — one turn covers the full tool loop) or get their own (cleaner per-generation, but inspector has to walk multiple turn_ids to see one user-facing turn)? **Default: separate turn_ids; the inspector summary groups by `parent_turn_id` if a future field lands.** Add `parent_turn_id` (nullable) to schema now to leave room.

3. **Content preview length.** 280 chars is a guess. Tweet-length, but tool results and code blocks can blow past it. Options: 280, 500, 1000. **Default: 500. Bigger preview is cheap; truncated content makes the trace less useful.**

4. **Metadata size cap.** `metadata_json` is unbounded by schema. Options: hard cap 4KB per record, soft cap with truncation on overflow, no cap. **Default: hard cap 4KB. Larger metadata is a producer bug.**

5. **Hash algorithm.** SHA-256 first 12 chars (~48 bits) is fine for collision detection within a 30-day window. Options: full sha256, sha256[:12], blake2b[:12]. **Default: sha256[:12] — collision-safe at this scale, fast on Windows.**

6. **`inspect_trace` tool naming and dispatch.** Option A: it's a `verb`-style tool (`{"name":"inspect_trace","arguments":{"verb":"recent","limit":5}}`) like advanced's `smart_system`. Option B: each query becomes its own tool name (`trace_recent`, `trace_one`, `trace_errors`). **Default: A (single tool, verb-dispatched). Reduces catalog bloat.**

7. **Failure isolation.** If a trace write fails (disk full, sqlite locked), does the turn fail? **Default: no — trace writes are best-effort, errors logged to stderr, turn proceeds. Trace is observation, not policy.**

## 8. Test surface

**Layer A producer tests** (`tests/test_turn_trace_stages.py`):
- StageTraceRecord invariants (kept/dropped, outcome/reason, time ordering).
- Each interceptor produces a record with the right `stage_kind`.
- Errored interceptor produces `outcome="errored"` with reason; outer error still propagates.
- Skipped interceptor produces `outcome="skipped"` with reason (e.g. flag off, no pins, already injected).

**Layer B producer tests** (`tests/test_turn_trace_frames.py`):
- FrameTraceRecord captures final messages with hashes + previews.
- Char counts match.
- Backend dispatch happens after the trace write (write doesn't block dispatch).

**Layer C query tests** (`tests/test_turn_trace_inspector.py`):
- `get_turn_trace` returns joined record for known turn_id.
- `get_turn_trace` returns None for unknown turn_id.
- `list_recent_turns` orders newest-first, respects limit.
- `search_turns(has_errored_stage=True)` filters correctly.
- `/trace` slash command parses correctly.
- `inspect_trace` tool returns structured data, not a stringified blob.

**Schema migration tests:**
- Running on a fresh DB creates the two tables.
- Running on a DB with old schema does not error.
- Adding a column via ALTER doesn't break old reads.

**Retention test:**
- TTL daemon deletes records older than threshold; preserves newer.

**Failure isolation test:**
- Trace write to a locked DB does not break the turn.

The user has not approved implementation. When/if approved, the order is:

1. **Storage primitives** — `core/turn_trace.py` with the SQLite schema, write paths, query API. No producers, no consumers wired. Tests for query logic against a hand-populated DB.

Estimated size: ~700 lines of code (core + tests + UI + tool), 4-6 hours of focused work.

## 10. Anti-theater check

If at any point during build the trace is producing data that no consumer reads, **stop**. The whole point of this spec is to NOT ship the bifurcated-orphan pattern. Layer C must be live before Layers A/B accumulate data, OR Layers A/B must be flag-off until C is live.

If at any point during build the trace mutates production behavior, **stop**. Trace is observation. The day a stage writes "I ran" and another stage reads that to decide whether to run, we've lost the diagnostic property.

## Appendix — Why the back half matters (and why we're not building it now)

The full causal failure loop the user named is:

```
input → stages (A) → frame (B) → response → outcome → verifier → memory_writes → next-turn input
```

Layers A + B + C cover the **input → frame** half. Without the back half, when something goes wrong you can answer "what did the model see" but not "what was wrong with what it produced" or "did the wrong thing get written to memory."

The back half requires:
- A user-action capture (did they accept / correct / redo / ignore)
- A verifier (Restore has none — would need the response_verifier subsystem from advanced)
- Memory-write hooks (continuity has writes; ACU has writes; chat archive has writes — each needs a side-channel emit)

Each of those is a separate subsystem with its own design questions. Combining them into Layer A/B/C now is exactly the vertical-without-horizontal mistake the synergy spec exists to prevent. Front half first, with hooks for back half preserved.
