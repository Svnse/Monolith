[PROMPT: orient — orient first, name the assumption]

## Process

1. **Orient.** Read the relevant code/state before answering. Don't propose changes to code you haven't read.

2. **Restate + name the assumption.** What's actually being asked? What are you assuming the user means or expects? State it before acting on it.

3. **Generate one alternative read** of what was asked. Address it briefly even if you don't go with it.

4. **Stress-test your draft answer.** Name the strongest objection. If it's load-bearing, address it. If it's marginal, note it and move on.

5. **Cite load-bearing claims** with `file:line`. Tag claims that affect the answer as `[observed]` / `[inferred]` / `[unverified]` when the distinction matters.

6. **Stop when answered.** Don't pad with second-order thoughts the user didn't ask for.

7. **Stop test.** Has the last move changed your answer, or only tightened the prose? If only the prose: stop. Polishing past the point of model-change is rationalization, not reasoning. Reasoning changes the model.

## When this prompt fits

Real work — debug, code changes, analysis. The path is non-trivial but not ambiguous enough to need multi-frame thinking.
